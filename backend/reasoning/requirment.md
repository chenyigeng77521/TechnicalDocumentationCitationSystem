# 技术文档智能问答与引用溯源系统挑战赛 · 选手数据包

> 版本:v1.0 — 正式发布版
> 发放日期:2026-04-28  规模:约 80 万字技术文档 · 3 个文档集(Kubernetes / Spring / React) · 200 条公开评测

---

## 1. 任务概述

构建一个**技术文档智能问答系统**,基于评委会提供的**公开技术文档集**回答评测集中的用户问题,每次回答必须附带至少一条**真实可追溯的出处**(文档路径 + 段落 anchor),并对文档中不存在答案的问题**明确拒答**。

考核重点(Context Engineering / RAG 基础能力):

- **多源文档处理**:语料覆盖 Kubernetes 官方文档、Spring Framework 文档、React 文档,格式包含 `.md`、`.adoc` 等
- **切分与索引**:文档需要你自己解析(含中英文混排、代码块、章节锚点)、切分、建索引
- **检索增强生成(RAG)**:基于检索结果生成准确、**可追溯到段落 anchor** 的答案
- **拒答能力**:对文档中不存在答案的问题,必须明确拒答而非编造
- **增量更新能力**:文档新增/修改/删除后,系统需在 **5 分钟内** 让变更生效
- **Web UI 交互**:提供可访问的 Web 界面供评委直接提问、查看答案与引用出处

子任务两类:**有答案题(150 题,75%) + 无答案陷阱题(50 题,25%)**

## 3. 文档集说明

### 3.1 文档来源与规模

| 语料源              | 格式                       | 主要内容                                |
| ---------------- | ------------------------ | ----------------------------------- |
| Kubernetes       | `.md`(中文为主,含代码块、表格、YAML) | 容器、架构、规则、计算-存储-网络、调度与驱逐、配置、集群管理     |
| Spring Framework | `.adoc` 为主,部分 `.md`      | Core、Web、WebFlux、Data、Testing 等模块文档 |
| React            | `.md`(英文/中文混合)           | 组件、Hooks、状态管理、性能优化、参考手册等            |

**全部为公开技术文档,本题不涉及任何内部知识库。** 三类文档总计约 **80 万字**。

### 3.2 引用出处:`doc_path` + `anchor`

每次回答必须给出**真实指向文档内容**的引用,引用格式为:

```json
{
  "doc_path": "docs/kubernetes/规则/resource-quotas.md",
  "anchor": "#本地临时存储的配额"
}
```

- `doc_path`:文档相对路径,从 `docs/` 开始,使用正斜杠
- `anchor`:文档内 H1/H2/H3/H4 标题对应的锚点(以 `#` 起始,中英文均可,空格转 `-`)
- 一题可有 **多条引用**,代表你的答案综合了多段内容

**这个 `(doc_path, anchor)` 二元组就是你提交时 `citations` 的取值**。

- 你的 RAG 系统检索到某段内容后,提取该段落所属的 `doc_path` 与最近一级标题作为 anchor
- 引用真实性会被自动化验证("假引用" → 该题视为错误回答),直接影响 **引用精确度** 评分

### 3.3 解析与切分注意事项

| 项                     | 说明                                                                  |
| --------------------- | ------------------------------------------------------------------- |
| Markdown 标题           | 推荐按 H2/H3 切分,每个 chunk 携带从 H1 到当前层级的标题路径                             |
| AsciiDoc(`.adoc`)     | Spring 文档大量使用 AsciiDoc,标题语法是 `=`/`==`/`===`,需用 `asciidoctor` 或自写解析器 |
| 代码块                   | 不要在三引号代码块中间切断;推荐保留完整代码块作为一个最小单元                                     |
| 中文标题 anchor           | 评测的参考 anchor 直接保留中文(如 `#本地临时存储的配额`),不要做拼音转写                         |
| Spring `.adoc` anchor | 形如 `#web-on-reactive-stack`,通常在文档头部 `[[...]]` 显式声明,优先采用             |

## 4. 评测集格式

`eval/Public_Test_Set.jsonl` 每行一个评测样本,统一含 `id` / `type` / `domain`,按 `type` 区分扩展字段,`Public_Test_Set.jsonl`格式如下

```json
{
    "id": "react_001",
    "domain": "React",
    "question": "React Compiler的增量采用是什么意思？",
    "is_answerable": true,
    "answer": "React Compiler可以增量采用，允许先在代码库的特定部分试用编译器。",
    "gold_sources": [
        {
            "doc_path": "docs/react/incremental-adoption.md",
            "anchor": "#top",
            "evidence": "React Compiler can be adopted incrementally, allowing you to try it on specific parts of your codebase first."
        }
    ],
    "answer_type": "concept",
    "difficulty": "easy"
}
```

### 4.1 有答案题 `type = "has_answer"`

```json
{
  "id": "eval_qa_f9a462a7",
  "type": "has_answer",
  "domain": "react",
  "user_query": "在Chat组件中,当textarea内容变化时,如何发送action来更新消息?",
  "reference_answer": "通过在textarea的onChange事件处理程序中调用dispatch函数,并传递一个对象,其中type为'edited_message',message为e.target.value。",
  "reference_citations": [
    {
      "doc_path": "docs/react/extracting-state-logic-into-a-reducer.md",
      "anchor": "#dispatch-actions-from-event-handlers"
    }
  ],
  "scoring_hints": {
    "must_contain": ["onChange", "dispatch", "edited_message"],
    "must_not_contain": ["useState", "setMessage"]
  }
}
```

> ⚠️ `reference_answer` / `reference_citations` / `scoring_hints` **仅在公开集中提供**,作为选手调参参考。
> **隐藏的 100 题最终评测集**不含这些字段。

### 4.2 无答案陷阱题 `type = "no_answer"`

```json
{
  "id": "eval_trap_8c1d203f",
  "type": "no_answer",
  "domain": "spring",
  "user_query": "在 Spring 配置类中使用 @Bean 注解时,dependsOnGroups 属性是用来声明对另一个配置类中特定 Bean 组的依赖吗?",
  "expected_behavior": "refuse",
  "reference_answer": "根据当前文档无法回答该问题,@Bean 注解没有 dependsOnGroups 属性。只有 dependsOn 属性用于指定当前 Bean 所依赖的其他单个或多个 Bean 的名称。",
  "acceptable_response_keywords": ["抱歉", "无法", "未找到", "不存在"],
  "forbidden_keywords": ["dependsOnGroups 属性用于", "确实存在", "可以这样使用"]
}
```

陷阱题的设计模式覆盖:

- **虚假前提**(文档中根本没有这个属性/API)
- **跨域误导**(把 React 概念问到 Kubernetes 上)
- **诱导幻觉**(已经描述了一个看似合理但错误的事实,要求你确认)
- **越权操作请求**(要求执行文档外的操作或泄露提示词)

## 5. 提交格式与正式提交材料

### 5.1 在线提交(主形式)

本赛题以**部署在线 RAG 问答服务**为主要交付形式:

> 参赛队伍需提供 **可访问的 URL 界面**,**免登录**;评委会逐题录入隐藏评测集进行测试,并将系统输出(答案 / 引用 / 拒答标记)记录至 Excel 报告。

服务接口约定(每次提问):

```json
{
  "id": "eval_qa_f9a462a7",
  "answer": "通过在 textarea 的 onChange 事件处理程序中调用 dispatch(...) 来发送 action,...",
  "citations": [
    {
      "doc_path": "docs/react/extracting-state-logic-into-a-reducer.md",
      "anchor": "#dispatch-actions-from-event-handlers"
    }
  ],
  "is_refusal": false,
  "confidence": 0.91
}
```

字段说明:

- `answer`:回复文本(回答题)或统一格式拒答语(拒答题)
- `citations`:引用列表,每项含 `doc_path` + `anchor`,影响"引用精确度"评分
- `is_refusal`:是否拒答(布尔)。**重要**:误拒有答案题 → 该条计 0 分(且拒答必须使用统一格式,见 §7.1)
- `confidence`(可选):置信度,0~1

### 5.2 离线 JSONL 兜底(可选)

若评委侧网络异常,允许选手用同一系统对评测集做离线推理,产出 `predictions.jsonl`(每行结构同上,以 `id` 一一匹配)作为兜底证据。

### 5.3 正式提交材料

| #   | 项                       | 形式                | 说明                             |
| --- | ----------------------- | ----------------- | ------------------------------ |
| 1   | 在线服务 URL                | 公网可访问链接           | 主交付形式;评委用于隐藏 100 题在线测评         |
| 2   | `predictions.jsonl`(兜底) | JSONL 文件          | 与评测集 `id` 一一匹配                 |
| 3   | 设计文档                    | Markdown / PDF    | 4-8 页,说明系统架构、关键设计、性能指标、创新点与复现性 |
| 4   | 代码仓库(可选)                | GitHub/GitLab URL | 用于复现性审计                        |

设计文档应描述:

- 系统架构与技术选型(解析 / 切分 / 索引 / 检索 / 生成 / 拒答 / 增量更新)
- 关键设计决策(尤其针对说明书 §7 的雷点)
- 性能指标(检索延迟、端到端延迟、增量更新时延)
- 创新点
- 复现性声明

## 6. 评分细则

### 6.0 总分构成

**最终总分(满分 100)= 客观项智能评估(80 分) + 主观项人工评估(20 分)**

| 类别          | 权重   | 评什么                | 怎么评           |
| ----------- | ---- | ------------------ | ------------- |
| **客观项智能评估** | 80 分 | 功能性 + 回答准确性        | 大模型助手 + 自动化校验 |
| **主观项人工评估** | 20 分 | UI/UX 设计 + 交互体验流畅度 | 评委人工评估        |

> 公开 200 题用于调参和自测参考;**最终排名以隐藏 100 题的一轮评测结果 + 主观项评估合并计算**。

---

### 6.A 客观项智能评估(满分 80)

#### 6.A.1 维度一:功能性评估(40 分)

| 评分项           | 分值   | 评分标准                                                                                         |
| ------------- | ----:| -------------------------------------------------------------------------------------------- |
| **UI 界面与交互**  | 20 分 | 基础:提供 Web UI,支持提问输入、答案展示、引用出处展示(10 分);进阶:界面美观、响应式、加载状态反馈(5 分);**多轮对话支持**(加分项):是否支持上下文追问(5 分) |
| **数据导入与增量更新** | 10 分 | 支持评委会文档集批量导入(5 分);**增量更新时效**:新增/修改/删除后 **5 分钟内**生效(5 分,超时扣分)                                 |
| **可观测性**      | 10 分 | 答案溯源、引用文章和段落级溯源(5 分);大模型请求链全节点可观测日志(5 分)                                                     |

#### 6.A.2 维度二:回答准确性(40 分)

设有答案题数为 N₁(隐藏集中为 75 题),无答案题数为 N₂(25 题)。

| 评分项             | 分值   | 计算方式                                                                              |
| --------------- | ----:| --------------------------------------------------------------------------------- |
| **回答准确率**(有答案题) | 20 分 | 基准 20 分,每错 1 题扣 `Y = 20 / N₁`;`得分 = 20 - 错误题数 × Y`,下限 0                           |
| **拒答准确率**(无答案题) | 10 分 | 基准 10 分,无答案题被错误回答 1 题扣 `Z = 10 / N₂`;`得分 = 10 - 错答题数 × Z`,下限 0                    |
| **引用精确度**       | 10 分 | 基准 10 分,自动化验证答案中引用是否真实指向文档内容,出现"假引用" 1 题扣 `F = 10 / N₁`;`得分 = 10 - 错误题数 × F`,下限 0 |

特殊规则:

- **无引用 / 引用格式错误**:有答案题即便回答正确,但缺路径或缺 anchor → 该题视为错误回答
- **拒答统一格式**:拒答必须使用统一表述(如"抱歉,我无法从提供的文档中找到答案"),格式不符按答错处理
- **假引用判定**:答案声称的事实与引用段落实际内容不一致(说 A 但引段落讲 B) → 该题计入引用精确度的错题
- **幻觉**:对有答案题给出文档外的内容 → 计入回答准确率的错题

> 客观项智能评估总分 = 功能性评估(40) + 回答准确性(40) = **80 分**

---

### 6.B 主观项人工评估(满分 20)

由评委人工打分,主要观察可访问的在线服务:

| 评分项            | 分值   | 评估维度与观察点                                                                              |
| -------------- | ----:| ------------------------------------------------------------------------------------- |
| **UI/UX 设计质量** | 10 分 | 美观度:配色、排版、字体符合技术产品审美(3 分);信息架构:答案、引用、置信度等信息层级清晰(4 分);细节处理:错误提示、空状态、加载动画(3 分)          |
| **交互体验流畅度**    | 10 分 | 响应速度:从提问到答案展示(网络正常)(4 分);操作便捷性:支持历史记录、收藏、复制答案(3 分);反馈机制:明确区分"有答案 / 无答案"状态,拒答理由清晰(3 分) |

**重要规则**:

- 文档**夸大事实 / 隐瞒外部依赖**(如未声明用了第三方 API)→ 相关维度记 0 分
- 文档与系统行为**严重不符**(如声称做了拒答优化但客观分拒答异常差)→ "可观测性" / "反馈机制" 维度记 0 分

## 7. 评分隐含规则与常见扣分点

### 7.1 一定要做对的事

| #   | 项                                              | 不做的代价                       |
| --- | ---------------------------------------------- | --------------------------- |
| 1   | 引用同时给出 `doc_path` 和 `anchor`,且 anchor 真实存在于文档中 | 视为错误回答,影响回答准确率 + 引用精确度      |
| 2   | 拒答时严格使用**统一格式**(包含"抱歉""无法从提供的文档中找到答案"等)        | 格式不符按答错处理,直接扣 `Z = 10/N₂` 分 |
| 3   | 文档变更后 **5 分钟内** 让索引生效(增量加索引/删索引/更新索引)          | 增量更新时效项扣分                   |
| 4   | 提供的 URL 评测期内**全程可访问**且**免登录**                  | 无法测评 → 该题视为错误回答             |
| 5   | 答案中**移除**或**替换**所有 `{{TEMPLATE_VAR}}` 占位符      | UI/UX 与交互体验项扣分              |
| 6   | Web UI 显式区分"有答案 / 拒答"状态,并展示引用出处                | 反馈机制扣分,UI 基础分扣              |

### 7.2 容易踩的 8 个雷

#### 雷 1:意图过严 → 误拒可答题 ⚠️⚠️⚠️

拒答正确性是 **gate** 类规则:有答案题被错误拒答,该题在"回答准确率"上直接计为错题(扣 `Y = 20/N₁` 分)。**比答错还痛**——因为答错有时还能在引用上不被判错;误拒会让你既丢回答分,也无法挣引用分。

#### 雷 2:全部拒答以刷拒答分 ⚠️⚠️⚠️

看似"拒答准确率"可以拉到 10 分满分,但有答案题(75 道)会全部被错拒,回答准确率 = 0,引用精确度 = 0,客观分总计可能不足 10 分。**反而比策略型选手低很多**。

#### 雷 3:引用 anchor 错位 / 没有 anchor ⚠️⚠️

- 错位:答案讲"本地临时存储",却引到了`#cpu-和-memory-资源配额` → 自动化对齐校验失败,计入"假引用"
- 缺 anchor:只给了 `doc_path` 没给 `#xxx` → 视为引用格式错误,该题作错答处理

#### 雷 4:中文 anchor 编码错误 ⚠️⚠️

中文标题在不同系统下生成 anchor 规则不同(GitHub / Hugo / docsify / 自建)。本题**统一以"原始中文 + 空格替换为 `-`"为准**(如 `#本地临时存储的配额`),不要拼音化、不要 punycode、不要去掉空格。

#### 雷 5:答案话术堆砌 ⚠️⚠️

评委侧人工读体验也会反映到主观分上。

- 反例:"亲爱的评委您好,非常感谢您的耐心等待,根据我系统的检索结果分析..."(还没到正文已 50 字)
- 正例:开门见山 + 核心结论 + 引用,通常 80-300 字

#### 雷 6:只 Top-1 检索,不召多个 ⚠️

有些问题需要综合 2-3 段。如果你只引一条,而参考引用是 2 条 → 引用精确度评分中 Recall 不足。**建议引用至少召回参考引用 ±1 条以内的数量**。

#### 雷 7:`.adoc` 解析丢锚点 ⚠️

Spring 大量 `.adoc` 文档使用 `[[xxx]]` 显式声明 anchor。常见错误:

- 把 `==` 当成 H2 但忽略了上一行的 `[[anchor-id]]` → anchor 全错
- 用通用 markdown 解析器跑 `.adoc` → 标题层级与代码块都识别错

**建议**:Spring 用 `asciidoctor`/`asciidoc3` 专用解析,React/K8s 用 markdown 解析。

#### 雷 8:增量更新不生效或晚生效 ⚠️

评委可能在评测中**当场新增/修改/删除一篇文档**,然后 5 分钟内提问验证。如果你的索引是 **build-once** 模式,这一项直接 0 分。**必须实现文件监听 + 增量索引(增 / 改 / 删)**。

### 7.3 不会被扣的事(放心做)

- ✅ 答案语言风格自由(口语/正式/分点/连贯都可)
- ✅ `citations` 顺序不影响评分
- ✅ `confidence` 字段可以不填或随便填
- ✅ 引用召回**多于**参考引用是 OK 的,只要每一条都真实可验证
- ✅ 同一答案可以引用跨文档(如同时引 K8s 和 Spring,只要内容相关)

### 7.4 防作弊红线 ⚠️⚠️⚠️

违反任一项 → 直接取消资格:

1. **隐藏 100 题严禁回流**:不得通过任何方式(prompt、网络请求、模型记忆、参数化)把隐藏评测集回流到训练数据
2. **不得把文档的章节锚点表预先 hash-lookup 后跳过检索**:必须走真实检索-生成-引用链路
3. **不得在评测期间对外发起额外抓取**:即在评测窗口内偷偷重新爬 K8s/Spring/React 官网充实知识
4. **不得调用未声明的外部服务**:仅允许使用规则允许或在设计文档中明确声明的模型与 LLM 网关

## 9. 推荐技术栈(仅供参考,不强制,鼓励创新)

文档解析:

- Markdown(Kubernetes / React):`markdown-it`、`mistune`、`unstructured.partition.md`,或自写 H2/H3 切分
- AsciiDoc(Spring):`asciidoctor`(Ruby)/ `asciidoc3`(Python),保留 `[[xxx]]` 显式 anchor
- 代码块保留:切分时不要切断 ```` ``` ```` 三引号块、YAML 块、`[source,xxx]` 块

切分:

- 按 H2/H3 标题切,每个 chunk 携带从 H1 到当前层级的标题路径作为元数据
- chunk 元数据保留 `(doc_path, anchor)` 二元组用于回填引用
- 滑窗或递归切分作为兜底

检索:

- 稠密向量:`bge-large-zh-v1.5` / `bge-m3`(K8s 中文文档友好)/ `e5-mistral`(英文 React 友好)
- 稀疏:BM25(`rank_bm25`,英文用空格分词,中文用 jieba)
- 混合 + 重排:`bge-reranker-v2-m3`

生成 + 拒答:

- LLM 调用走主办方网关或自部署开源模型
- **拒答触发器**(三选一或组合):
  1. 检索分数低于阈值(向量相似度 + BM25 score 双阈值)
  2. 重排后 Top-1 仍低于阈值
  3. 让 LLM 在生成前先做 "is_answerable" 二分类

增量更新:

- `watchdog` 监听文件变更 → 增量解析 → 增量更新向量库 + BM25 倒排索引
- 删除事件需同步删除向量与倒排,避免 stale 引用

Web UI:

- 前端:Next.js / SvelteKit / Vue 任一;明确展示"答案 / 引用列表 / 拒答状态 / 加载动画"
- 多轮加分项:本地 session 维护对话历史,前端把历史 turns 一并提交后端

可观测性:

- OpenTelemetry / LangSmith / 自建日志:记录 retrieve 候选、rerank 分数、prompt、LLM 输出、最终引用
