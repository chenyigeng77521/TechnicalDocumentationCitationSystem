# 技术文档智能问答与引用溯源系统

> **赛题背景**
> 
>     "能否基于一批文档回答问题、且准确引用出处、不知道就说不知道"，是 Context Engineering 最基本的考查点。本题用公开技术文档作为语料，不涉及任何内部知识库。
> 
> **核心任务**
> 
> 1. 对评委会给定的文档集构建检索问答系统
> 2. 每次回答必须附带至少一条出处（文档路径 + 段落 anchor）
> 3. 对文档中不存在答案的问题，必须明确拒答而非幻觉
> 4. 支持文档增量更新（新增/修改/删除后 5 分钟内生效）
> 5. 提供 WebUI 供评委交互

## 1. 数据处理层

> 👉 负责：**“原材料对不对”**
> 
> - 文档解析错了（PDF 少页 / 乱码）
> - chunk 切错（一句话被拆烂）
> - embedding 明显异常（全是一样的向量）
> - 新文档上传后查不到
> 
> 👉 **“这玩意一开始就不对”** → 找数据层

### 1.1 文档解析

**设计**：按扩展名 + MIME sniff 路由到对应解析器，统一输出 `ParseResult`。MVP 全 Python 实现，所有格式 `content_type` 固定为 `"document"`（代码类感知切分 P1，详见 §1.1.3）。

#### 1.1.1 支持格式与解析器路由

**文档类** ：

| 扩展名 | 解析器 | Python 包 | 处理方式 | 优先级 |
| --- | --- | --- | --- | --- |
| `.md` | markdown 直读 | `markdown` | 提取 heading 作 title_tree | P0 |
| `.txt` | 直读 + 编码检测 | `chardet` | utf-8 / gbk 自动 | P0 |
| `.html` | 转 markdown | `beautifulsoup4` + `markdownify` | — | P0 |
| `.pdf`（文字版） | PyMuPDF | `pymupdf` | 检测 `page.get_text()` 非空 | P0 |
| `.pdf`（扫描版） | PaddleOCR | `paddleocr` | 文字提取为空时降级 | P1 |
| `.docx` | python-docx → md | `python-docx` | 保留 heading | P0 |
| `.doc`（旧版） | LibreOffice 转 .docx → python-docx | `unoconv` | MVP 直接报 `unsupported_format` | P1 |
| `.xlsx` | openpyxl → md 表格 | `openpyxl` | 每 sheet 一个段落 | P0 |
| `.xls`（旧版） | LibreOffice 转 .xlsx → openpyxl | — | MVP 直接报 `unsupported_format` | P1 |
| `.pptx` | python-pptx → md | `python-pptx` | 每 slide 一个段落 | P1 |
| `.ppt`（旧版） | LibreOffice 转 .pptx → python-pptx | — | MVP 直接报 `unsupported_format` | P1 |

**代码类（MVP 按 document 处理，不走感知切分）** ：

> ⚠️ **MVP 已敲定**：以下所有代码类文件一律按 `document` 处理，不走 AST / 结构感知切分。`content_type` 固定 `"document"`。P1 实现 §1.2.2 code 类切分策略后再启用。

| 扩展名 | 语言  | MVP 处理方式 | P1 规划 |
| --- | --- | --- | --- |
| `.py` | Python | 按 document 三级 fallback 切 | `^class` / `^def` / `^async def` 切 |
| `.java` | Java | 按 document 三级 fallback 切 | 按 class / method 切 |
| `.cpp` / `.hpp` / `.h` | C/C++ | 按 document 三级 fallback 切 | 按函数定义切 |
| `.sql` | SQL | 按 document 三级 fallback 切 | 按 `;` + 空行切 |
| `.json` | JSON | 按 document 三级 fallback 切 | 小文件 1 chunk；大文件按顶层 key 切 |
| `.yaml` / `.yml` | YAML | 按 document 三级 fallback 切 | 按顶层 key 切；保留注释 |
| `.xml` | XML | 按 document 三级 fallback 切 | 按根 tag 切 |

#### 1.1.2 统一输出格式

> **实现说明**：MVP 采用全 Python（`backend/ingestion/` FastAPI :3003）统一处理所有格式，不再区分 Node/Python 双栈。`ParseResult` 为 Python dataclass：

```python
@dataclass
class ParseResult:
    raw_text: str                    # 解析后纯文本
    title_tree: list[TitleNode]      # heading 层级，可空
    content_type: str = "document"   # MVP 全部固定 document（见 §1.1.3）
    language: str | None = None      # zh / en / auto
    confidence: float = 1.0          # 解析置信度 0-1（OCR 时填实际值）
    metadata: dict = field(default_factory=dict)  # 解析器特有字段（pdf_pages / pdf_is_scanned / sheet_names 等）
```

> **解析器对应关系（Python 实现）**：PDF 文字版 → `PyMuPDF`；扫描版 → `PaddleOCR`；docx → `python-docx`；xlsx → `openpyxl`；pptx → `python-pptx`；md/txt/html → 内置库。

#### 1.1.3 关键决策

- ✅ **全 Python 单栈解析**（`backend/ingestion/` FastAPI :3003，取消 Node 双栈，简化部署）
- ✅ **LibreOffice 统一旧版 Office 转换**（`.doc/.xls/.ppt` MVP 直接报 `unsupported_format`，P1 补齐）
- ✅ **MIME sniff 兜底**：`python-magic` 二次确认真实类型，扩展名不可信时覆盖分派
- ✅ **代码不加 OCR**（源码不可能是扫描件）
- ✅ **代码类文件不走 AST 切分（MVP 已敲定）**：`.py / .java / .json / .yaml` 等代码类文件全部按 `document` 处理，`content_type` 固定 `"document"`。`code` / `structured_data` 两套 chunking 策略 **MVP 不实现**，推迟到 P1（对应 §1.2.2 / §1.2.3 中的设计保留作为 P1 规格）

### 1.2 Chunk 切分策略

> **MVP 范围**：MVP 仅实现 **document 类型**三级 fallback 切分（§1.2.1）。`code`（§1.2.2）和 `structured_data`（§1.2.3）切分策略保留设计规格，**P1 实现**。MVP 阶段所有文件均走 document 路径，`content_type` 固定 `"document"`。

按 `content_type` 分派到三套切分策略（Python 实现，位于 `backend/ingestion/chunker/`）。

#### 1.2.1 document 类型：段落 → 句子 → 硬切 三级 fallback

适用于 md / pdf / docx / pptx / html / txt 转出的 markdown：

```
第 1 级：按标题（title_tree）切成大段
   │  段落 ≤ MAX_CHARS → 直接作为 chunk
   ▼
第 2 级：按段落边界（\n\n）切分
   │  段落 ≤ MAX_CHARS → 直接作为 chunk
   ▼
第 3 级：按句子边界（。！？. ! ?）切分
   │  单句仍 > MAX_CHARS → 触发硬切（is_truncated: true）
   ▼
添加 200 char overlap（取前一 chunk 末尾拼到下一 chunk 前面）
```

#### 1.2.2 code 类型：语言感知的结构化切分（P1，MVP 不实现）

> ⚠️ **P1 规格**：MVP 阶段代码类文件走 document 路径，本节为 P1 实现目标。

```
第 1 级：按语言的"顶级定义"切（class / def / function / CREATE 等）
   │  block ≤ MAX_CHARS → 直接作为 chunk，函数名/类名作为 title_path
   ▼
第 2 级：超长函数按空行切逻辑块
   │  逻辑块仍 > MAX_CHARS → 触发硬切（极少见）
   ▼
注意：代码 chunk 不加 overlap（语法上会导致重复）
```

各语言顶级定义匹配规则：

| 语言  | 切分边界 |
| --- | --- |
| Python | `^class` / `^def` / `^async def` |
| Java | `public/private/protected … class/interface/enum` |
| C/C++ | `class … {` / 函数定义 |
| SQL | `CREATE / SELECT / INSERT / UPDATE / DELETE` |
| 未知语言 | 整文件一个 block（兜底） |

#### 1.2.3 structured_data 类型：按顶层 key / 节点切（P1，MVP 不实现）

> ⚠️ **P1 规格**：MVP 阶段 JSON/YAML/XML 文件走 document 路径，本节为 P1 实现目标。

- 文件 ≤ MAX_CHARS → 整体 1 chunk
- JSON / YAML → 按顶层 key 切（每个 key 独立 chunk，title_path 用 key 路径）
- XML → 按根 tag 的直接子节点切
- 解析失败 → 降级为纯文本硬切

#### 1.2.4 Chunk 数据结构

```typescript
interface Chunk {
  chunk_id: string;        // sha256(file_path + chunk_index + content[:100])
  file_path: string;
  file_hash: string;
  content: string;
  anchor_id: string;       // "file_path#char_offset_start"
  title_path: string | null;
  char_offset_start: number;
  char_offset_end: number;
  char_count: number;      // 用 char 数代替 token 数（避免 tokenizer 不对齐）
  is_truncated: boolean;   // 硬切时 true，Layer 3 在 prompt 中注明
  chunk_index: number;
  content_type: "document" | "code" | "structured_data";
  language?: string;
  embedding?: number[];
}
```

**Anchor 设计（双字段，职责分离）：**

- `anchor_id`（主键，程序定位用）：`file_path#char_offset_start`，例如 `api/auth.md#4821`（**相对 `backend/storage/raw/` 的路径，不带 `raw/` 前缀**）
- `title_path`（可读性附属字段，UI 展示用）：`Authentication > OAuth2 > Token Refresh`

对于无标题的文档（纯文本、配置文件等），`title_path` 留空，仅使用 `anchor_id`，系统不因此降级。

**各类型关键决策汇总：**

| 决策项 | document | code | structured_data |
| --- | --- | --- | --- |
| 切分粒度 | 段落/句子/硬切 | 顶级定义/逻辑块 | 顶层 key |
| overlap | 200 char | **无** | 无   |
| title_path 来源 | heading 路径 | 函数名/类名 | key 路径 |
| 硬切 fallback | ✓（罕见） | ✓（极少） | ✗（按 parser） |
| 质量过滤（< 30 char） | ✓   | ✗（短函数也有意义） | ✗   |
| 保留原始格式 | 允许归一化 | **严格保留** | **严格保留** |

### 1.3 存储

**SQLite + FTS5 + 向量列**

**Schema**：

> **与 spec 对齐说明**：`index_version` MVP 写死 `'v1'`，schema 留位供未来 chunk_diff 原子切换使用；`CREATE TABLE` 均加 `IF NOT EXISTS` 保证幂等；触发器名加 `IF NOT EXISTS`（SQLite 3.31+）。

```sql
PRAGMA journal_mode = WAL;
PRAGMA foreign_keys = ON;

-- 文档级元数据
CREATE TABLE IF NOT EXISTS documents (
    file_path        TEXT PRIMARY KEY,
    file_name        TEXT NOT NULL,
    file_hash        TEXT NOT NULL,
    file_size        INTEGER NOT NULL,
    format           TEXT NOT NULL,
    language         TEXT,
    index_version    TEXT NOT NULL DEFAULT 'v1',  -- UUID。MVP 固定 'v1'，schema 留位供原子切换
    index_status     TEXT DEFAULT 'pending',  -- pending / indexed / error
    error_detail     TEXT,
    chunk_count      INTEGER DEFAULT 0,
    last_modified    TIMESTAMP NOT NULL,
    indexed_at       TIMESTAMP,
    created_at       TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Chunks 主存储
CREATE TABLE IF NOT EXISTS chunks (
    chunk_id          TEXT PRIMARY KEY,
    file_path         TEXT NOT NULL,
    file_hash         TEXT NOT NULL,
    index_version     TEXT NOT NULL,
    content           TEXT NOT NULL,
    anchor_id         TEXT NOT NULL,
    title_path        TEXT,
    char_offset_start INTEGER NOT NULL,
    char_offset_end   INTEGER NOT NULL,
    char_count        INTEGER NOT NULL,
    chunk_index       INTEGER NOT NULL,
    is_truncated      INTEGER DEFAULT 0,
    content_type      TEXT NOT NULL DEFAULT 'document', -- document / code / structured_data
    language          TEXT,                             -- zh / en / python / java / ... 与 documents.language 同
    embedding         TEXT,             -- JSON array（1024 dims）
    created_at        TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (file_path) REFERENCES documents(file_path) ON DELETE CASCADE
);

CREATE INDEX IF NOT EXISTS idx_chunks_file    ON chunks(file_path);
CREATE INDEX IF NOT EXISTS idx_chunks_version ON chunks(index_version);

-- FTS5 全文索引（BM25）
CREATE VIRTUAL TABLE IF NOT EXISTS chunks_fts USING fts5(
    chunk_id UNINDEXED,
    content,
    title_path,
    tokenize = 'unicode61 remove_diacritics 2'
);

-- 自动同步触发器
CREATE TRIGGER IF NOT EXISTS chunks_ai AFTER INSERT ON chunks BEGIN
    INSERT INTO chunks_fts(chunk_id, content, title_path)
    VALUES (new.chunk_id, new.content, new.title_path);
END;

CREATE TRIGGER IF NOT EXISTS chunks_ad AFTER DELETE ON chunks BEGIN
    DELETE FROM chunks_fts WHERE chunk_id = old.chunk_id;
END;

CREATE TRIGGER IF NOT EXISTS chunks_au AFTER UPDATE ON chunks BEGIN
    DELETE FROM chunks_fts WHERE chunk_id = old.chunk_id;
    INSERT INTO chunks_fts(chunk_id, content, title_path)
    VALUES (new.chunk_id, new.content, new.title_path);
END;
```

### 1.4 增量同步

**设计**：双路径 → 单 Pipeline → 文件级锁 → **MVP 全删重写** → （P1 升级：chunk 级 diff → 原子版本切换）

增量触发来源有两条路径，统一合并到同一 Pipeline：

```
路径 A：entrance 上传触发（同步）
  entrance multer 落地 backend/storage/raw/ → POST :3003/index { file_path } → index_pipeline()

路径 B：外部文件变更（评委直接拷贝文件到目录，兜底）
  watchdog 监听 backend/storage/raw/ → debounce 1s → index_pipeline()
```

#### MVP 实现（全删重写）

> ⚠️ **MVP 降级说明**：MVP 采用"先删旧 chunks 再写新 chunks"策略，不实现 chunk_diff 和原子版本切换（见"不做"清单）。`index_version` 字段保留、固定 `'v1'`，schema 为 P1 原子切换留位。大文档更新会重算所有 chunk 的 embedding（无 embedding 复用），且有短暂"无旧数据-有新数据"过渡窗口（delete → insert 之间）。

`index_pipeline` MVP 完整流程：

```python
async def index_pipeline(file_path: str) -> dict:
    """
    路径 A 和路径 B 共用的入口。
    file_path 是相对 backend/storage/raw/ 的路径（如 "example.pdf"）。
    """
    abs_path = resolve_under_storage_raw(file_path)
    if not abs_path.exists():
        raise FileNotFoundError(file_path)

    async with file_lock(file_path):  # 文件级锁，防 A/B 并发
        # 1. file_hash；hash 未变直接返回
        new_hash = sha256_of_file(abs_path)
        old_doc = db.get_document(file_path)
        if old_doc and old_doc.file_hash == new_hash:
            return {"status": "unchanged"}

        # 2. upsert documents 行（status=pending）
        new_version = "v1"  # MVP 固定
        db.upsert_document(file_path=file_path, file_hash=new_hash,
                           file_size=abs_path.stat().st_size,
                           format=abs_path.suffix.lstrip("."),
                           index_version=new_version,
                           index_status="pending",
                           last_modified=datetime.utcnow())

        try:
            # 3. 解析
            parse_result = await parse_document(abs_path)

            # 4. 切 chunk
            chunks = chunker.split(parse_result, file_path=file_path,
                                   file_hash=new_hash, index_version=new_version)

            # 5. 算 embedding（并发 8，本地 bge-m3 无 rate limit 限制）
            embeddings = await batch_embed(
                texts=[c.content for c in chunks],
                concurrency=8,
            )
            for c, emb in zip(chunks, embeddings):
                c.embedding = emb

            # 6. 写库（事务）：MVP 全删重写
            with db.transaction():
                db.delete_chunks_by_file(file_path)  # 删旧 chunks（CASCADE 删 fts）
                db.insert_chunks(chunks)             # 写新 chunks（触发器同步 fts）
                db.update_document(file_path=file_path,
                                   index_status="indexed",
                                   chunk_count=len(chunks),
                                   indexed_at=datetime.utcnow(),
                                   error_detail=None)

            return {"status": "indexed", "chunk_count": len(chunks), "file_hash": new_hash}

        except ParseError as e:
            db.update_document(file_path=file_path, index_status="error",
                               error_detail=f"解析失败: {e}")
            raise
        except EmbeddingError as e:
            db.update_document(file_path=file_path, index_status="error",
                               error_detail=f"embedding 失败: {e}")
            raise
```

**删除处理**：

```python
async def handle_file_delete(file_path: str):
    async with file_lock(file_path):
        db.delete_document(file_path)  # ON DELETE CASCADE 删所有 chunks
```

**孤儿 GC**（启动时 + 每小时）：

```python
async def initial_scan_and_gc():
    disk_files = set(walk_raw_dir())
    db_files = set(db.get_all_document_paths())
    for fp in disk_files - db_files:
        asyncio.create_task(index_pipeline(fp))
    for fp in db_files - disk_files:
        asyncio.create_task(handle_file_delete(fp))

async def hourly_gc():
    while True:
        await asyncio.sleep(3600)
        await initial_scan_and_gc()
        db.execute("""
            DELETE FROM chunks
            WHERE file_path NOT IN (SELECT file_path FROM documents)
        """)
```

**MVP 增量场景对照表**：

| 场景 | 行为 |
|---|---|
| 文件首次上传 | 走完整 pipeline，写 documents + chunks |
| 同一文件再上传，hash 没变 | 返回 `unchanged`，不重新处理 |
| 同一文件再上传，hash 变了 | 删 file_path 下所有旧 chunks → 走完整 pipeline → 写新 chunks |
| 文件被删（路径 B 监听到 deleted 事件） | 删 documents 行（CASCADE 删 chunks / fts） |
| 启动时全扫 GC | 对比 raw/ 实际文件 vs documents 表，缺失的删，多余的触发 index |
| 每小时定时 GC | 删孤儿 chunks（没有对应 document 的） |

**MVP 不做（明确写出来，避免歧义）**：

- ❌ **chunk-level diff（embedding 复用）**：P1 实现，届时参照下方 P1 规格
- ❌ **index_version 原子切换**：schema 字段保留，MVP 始终写 `'v1'`，P1 启用
- ❌ **chunk-level 软删除**
- ❌ **WebSocket 进度推送**：MVP 同步等返回，P1 补推送（详见 §1.5）

#### P1 规格（chunk_diff + 原子切换，供未来参考）

> 以下为 P1 完整方案，当前不实现。

```python
# P1: chunk_diff 算法（embedding 复用省 99% 调用）
def chunk_diff(new_chunks, old_chunks):
    new_ids = {c.chunk_id: c for c in new_chunks}
    old_ids = {c.chunk_id: c for c in old_chunks}
    to_add    = [c for id, c in new_ids.items() if id not in old_ids]
    to_delete = [c for id, c in old_ids.items() if id not in new_ids]
    reuse     = [(new_ids[id], old_ids[id]) for id in new_ids if id in old_ids]
    return to_add, to_delete, reuse
```

**关键决策（已确认）：**

- ✅ **文件级锁**：`asyncio.Lock`（Python），单进程够用，不需要 Redis 分布式锁
- ✅ **debounce 1 秒**：抑制文件事件风暴（快速连续写入合并为一次）
- ✅ **watchdog 监听**：路径 B 兜底，debounce 后与路径 A 共用 `index_pipeline`
- ✅ **定时 GC**：防漏事件导致孤儿 chunks 堆积
- ⏳ **chunk_diff 复用 embedding**：P1，大文档改一段时省 99% embedding 调用
- ⏳ **原子版本切换**：P1，查询永远看到一致版本，失败保留老数据

### 2.1 查询扩展（Query Expansion）

**问题背景：** bge-m3 在以下场景存在语义理解局限：

- **专业术语漂移**：`RAFT`、`gRPC`、`CRD` 等技术专有名词的向量表示可能和通用语义混淆，导致召回偏离
- **短查询语义稀疏**：评委提问往往很短（如"超时配置"），短文本 embedding 向量信息密度低，召回方差大
- **中英混合对齐偏差**：技术文档中文描述 + 英文术语混排，跨语言语义对齐得分偏低

**实现方案（三路扩展）：** retrieval.py `expand_query()` 实现三路扩展，每路独立双路检索，最后合并去重：

```python
def expand_query(query: str, num_variants: int = 3) -> List[str]:
    # 1. LLM 同义变体（最多 num_variants 个，超出截断）
    variants = call_llm(
        f"请为以下查询生成 {num_variants} 个语义等价查询，每行一个，不加编号：\n{query}"
    )  # 如：超时配置 → ["连接超时设置", "timeout 配置方法"]

    # 2. Bing 跨语言翻译（中文→英文，增强英文文档召回）
    english_text = ts.translate_text(query, translator='bing',
                                     from_language='zh', to_language='en')
    # 3. 原查询保留在首位
    all_queries = [query] + variants[:num_variants] + [english_text]
    return all_queries[:num_variants + 2]  # 最多 5 个（原始 + 3 变体 + 英译）

# 示例：
# 原始查询：  "超时配置"
# 扩展后：    ["超时配置", "连接超时设置", "timeout 配置方法", "timeout configuration"]
```

**每变体独立检索：**

```python
# 每个查询变体独立走双路检索，合并结果
```

### 2.2 两路召回 + 合并去重

**架构变更（retrieval.py → ingestion :3003）：**

retrieval 服务**不直连 SQLite**，双路检索全部走 HTTP API 调用 ingestion 层：

```
retrieval.py                         ingestion (:3003)
  │                                     │
  │  POST /chunks/vector-search         │
  │  body: { embedding: [...], top_k }  │
  │────────────────────────────────────►│
  │                                     │  bge-m3 计算 query embedding（本地）
  │                                     │  cosine similarity 查询
  │◄────────────────────────────────────│
  │  { results: [{ chunk_id, score,     │
  │              content, metadata }] } │
  │                                     │
  │  POST /chunks/text-search           │
  │  body: { query: str, top_k }        │
  │────────────────────────────────────►│
  │                                     │  FTS5 BM25 MATCH
  │◄────────────────────────────────────│
  │  { results: [{ chunk_id, score,     │
  │              bm25_rank, metadata }] │
  │                                     │
```

> 环境变量 `VECTOR_API_URL` 配置 ingestion 服务地址，默认 `https://equivalent-handling-heritage-hat.trycloudflare.com/`（Cloudflare 代理）。

**路 1：向量检索 API（POST /chunks/vector-search）**

- retrieval 本地用 bge-m3 计算 query embedding（`normalize_embeddings=True`）
- 发送 embedding 向量到 ingestion 向量检索端点
- 阈值过滤：`score < VECTOR_SCORE_THRESHOLD`（默认 0.55，环境变量可配）

**路 2：BM25 检索 API（POST /chunks/text-search）**

- 直接传字符串，不在 retrieval 侧计算 embedding
- ingestion 层执行 FTS5 BM25 MATCH
- 阈值：`score < BM25_SCORE_THRESHOLD`（默认 -999.0，**不过滤**）

**合并去重（`_merge_results`）：**

```python
def _merge_results(vec_docs, bm25_docs) -> List[Document]:
    """按 chunk_id 去重，保留双路分数（vec score + bm25 score）"""
    all_docs = {}
    for doc in vec_docs:
        key = doc.metadata.get("chunk_id", doc.page_content[:100])
        all_docs[key] = doc                    # 首次出现保留向量分数
    for doc in bm25_docs:
        key = doc.metadata.get("chunk_id", doc.page_content[:100])
        if key in all_docs:
            all_docs[key].metadata["bm25_rank"] = doc.metadata.get("bm25_rank")
            all_docs[key].metadata["bm25_score"] = doc.metadata.get("score")
        else:
            all_docs[key] = doc                # 仅 BM25 命中的文档
    return list(all_docs.values())
```

> 注：retrieval.py 中两路合并采用**去重合并**，非 RRF 公式。RRF 融合方案（加权融合 + rank 倒数求和）作为 P1 规格在下方保留供参考。

**P1 RRF 融合规格（保留设计）：**

> 以下为未来升级 RRF 融合的参考方案，当前不实现。

```python
# P1: RRF 融合（k=60，控制头部权重）
def rrf_fusion(vec_docs, bm25_docs, k=60):
    # 1. 分别建立 rank 映射（按分数降序）
    bm25_ranks  = {d.metadata["chunk_id"]: r+1 for r, d in enumerate(sorted(bm25_docs, key=lambda d: d.metadata.get("bm25_score", 0), reverse=True))}
    dense_ranks = {d.metadata["chunk_id"]: r+1 for r, d in enumerate(sorted(vec_docs, key=lambda d: d.metadata.get("score", 0), reverse=True))}
    # 2. 所有 chunk 并集，按 RRF 公式合并
    all_ids = set(bm25_ranks) | set(dense_ranks)
    rrf_scores = {}
    for cid in all_ids:
        r1 = bm25_ranks.get(cid, float('inf'))
        r2 = dense_ranks.get(cid, float('inf'))
        rrf_scores[cid] = 1 / (k + r1) + 1 / (k + r2)
    # 3. 截取 Top 20
    return sorted(rrf_scores.items(), key=lambda x: x[1], reverse=True)[:20]
```

### 2.3 自适应 TopK

根据问题类型用规则动态调整 TopK，范围 `[ADAPTIVE_TOPK_MIN=5, ADAPTIVE_TOPK_MAX=25]`：

```python
def adaptive_topk_simple(query: str) -> int:
    k = 10                                   # 基础值

    # 规则1：长查询（>30字符）需要更多上下文
    if len(query) > 30:
        k += 5

    # 规则2：多关键词（空格>8个）表示复杂问题
    if query.count(' ') > 8:
        k += 5

    # 规则3：含数字（版本号、代码、参数值）需要多召回
    if any(c.isdigit() for c in query):
        k += 3

    # 规则4：疑问句/比较类问题扩大召回
    if any(word in query for word in ['如何', '为什么', '对比', '区别']):
        k += 2

    # 规则5：技术文档专项（算法/实现/架构/原理/优化）
    tech_patterns = ['算法', '实现', '架构', '原理', '优化']
    if any(p in query for p in tech_patterns):
        k += 3

    return min(max(k, ADAPTIVE_TOPK_MIN), ADAPTIVE_TOPK_MAX)
```

> `ADAPTIVE_TOPK_MIN` / `ADAPTIVE_TOPK_MAX` 通过环境变量配置（默认 5 / 25）。

### 2.4 Reranker 精排与过滤

使用 `bge-reranker-v2-m3` 对合并后结果重新打分，支持**本地 CrossEncoder** 和**远程 API** 两种模式：

**双模式切换（环境变量控制）：**

```python

# RERANK_API_URL 配置 → 远程 API 重排序

```

**本地 CrossEncoder 模式：**

```python
class Reranker:
    def __init__(self, model_name="BAAI/bge-reranker-v2-m3", top_n=5):
        self.model = CrossEncoder(
            model_name,
            tokenizer_kwargs={'truncation': True, 'padding': True}
        )
        self.top_n = top_n

    def rerank(self, query: str, docs: List[Document]):
        # 1. 上下文扩展：同文件相邻 chunk 拼接（window=1，前后各1个）
        expanded_texts = _expand_rerank_context(docs, window=1)
        # 2. CrossEncoder 打分（logits → Sigmoid 映射到 (0,1)）
        scores = self.model.predict(pairs=[[query, t] for t in expanded_texts], batch_size=8)
        scores = 1 / (1 + np.exp(-np.array(scores)))  # Sigmoid 映射
        # 3. 分数写入 metadata，供下游阈值判断
        for idx, score in enumerate(scores):
            docs[idx].metadata["reranker_score"] = float(score)
        # 4. 降序截断 top_n
        return [docs[i] for i in np.argsort(scores)[::-1][:self.top_n]]
```

**远程 API 模式：**

```python
class APIReranker:
    def rerank(self, query: str, docs: List[Document]):
        expanded_texts = _expand_rerank_context(docs, window=1)
        response = requests.post(
            RERANK_API_URL,          # 默认 https://aigw.asiainfo.com/v1/rerank
            json={
                "model": RERANK_API_MODEL,   # 默认 10086/bge-reranker-v2-m3
                "query": query,
                "documents": expanded_texts,
                "return_text": True
            }
        )

                        key=lambda x: x.get("relevance_score", 0), reverse=True)
        # 按 relevance_score 降序截取 top_n
        for item in results[:self.top_n]:
            idx = item["index"]
            docs[idx].metadata["reranker_score"] = item["relevance_score"]
        return [docs[r["index"]] for r in results[:self.top_n]]
```

**上下文扩展（`_expand_rerank_context`）：**

```python
def _expand_rerank_context(docs, window=1) -> List[str]:
    """同文件内按 char_offset 排序，相邻 window 个 chunk 拼接后再打分"""
    by_file = {}
    for doc in docs:
        by_file.setdefault(doc.metadata.get("file_path", ""), []).append(doc)
    for fp in by_file:
        by_file[fp].sort(key=lambda d: d.metadata.get("char_offset_start", 0))
    # 对每个 doc，取其相邻 window 个 chunk 内容拼接
    expanded = []
    for doc in docs:
        fp = doc.metadata.get("file_path", "")
        file_docs = by_file.get(fp, [])
        idx = next((i for i, d in enumerate(file_docs) if id(d) == id(doc)), None)
        if idx is None:
            expanded.append(doc.page_content)
            continue
        parts = [file_docs[j].page_content
                 for j in range(max(0, idx-window), min(len(file_docs), idx+window+1))]
        expanded.append("\n".join(parts))
    return expanded
```

**精排后处理流水线：**

```
合并去重结果（来自 §2.2）
          │
          ▼
  bge-reranker-v2-m3 精排
 （Sigmoid 映射到 (0,1)，分数写入 reranker_score）
          │
          ▼
  按 reranker_score 降序排列，截取 RERANK_TOP_N（默认 5）
          │
  ┌───────┴──────────────────┐
  │  max_score < 0.4？       │   ← 硬性门控，代码判断，不经过 LLM
  │       YES                │
  └───────┬──────────────────┘
          │ NO
          ▼
  硬截断保留 token < 6k
  （从高分 chunk 开始保留，截断时在 prompt 中注明 "context 可能不完整"）
          │
          ▼
    进入 LLM 推理
```

> **Sigmoid 映射说明**：`bge-reranker-v2-m3` 输出 logits，retrieval.py 用 `1/(1+exp(-x))` 映射到 (0,1)，与 `bge-reranker-base` 输出行为对齐，保证分数阈值 0.4 的语义一致性。

## 4. Web UI 层

> 👉 负责：**“展示 / 交互 / 状态”**
> 
> - UI 显示不对（引用点错位置）
> - 进度条不更新
> - 接口返回乱
> - WebSocket 不推送
> 
> 👉 **“东西是对的，但你没展示对” → 找前端**

前端 `Next.js`，后端 `FastAPI` / Node Express

### 布局

```
┌─────────────────┬──────────────────────────────┬──────────────────┐
│   文档管理面板   │         对话区                │  Chunk 预览面板  │
│                 │                              │  （可折叠）       │
│ ▸ auth.md  ✓   │  Q: OAuth2 如何刷新 Token？   │ [Chunk #1]       │
│ ▸ deploy.md ✓  │                              │ Source: auth.md  │

│                 │  默认 7 天 [2]。             │ ──────────────── │
│ [+ 上传文档]    │                              │ [Chunk #2]       │
│                 │  引用列表 ▼                  │ Source: ...      │

└─────────────────┴──────────────────────────────┴──────────────────┘
```

### 关键交互细节

- **引用链接**：点击 `[1]` 跳转原文，使用 `anchor_id` 精确定位（`file_path#char_offset`），展示 `title_path` 作为可读文本
- **可信度标记**：`✓` / `?` 异步回填，初始显示为灰色，验证完成后更新
- **错误状态**：`✗ error` 展示具体原因（如"PDF 解析失败，请确认文件未加密"），可点击重试
- **进度展示**：`pending` 状态通过 WebSocket 实时显示阶段（parsing / chunking / embedding）+ 百分比 + 预计剩余时间
- **拒答展示**：拒答响应附带完整调试信息，供评委验证检索质量：
  - 拒答类型：`debug_info.trap_type`（8种：fake_api / future_version / overgeneralization / parameter_mismatch / cross_domain / concept_confusion / procedure_step / non_existent_attribute）
  - 拒答原因：`debug_info.unanswerable_reason`（LLM 生成的自然语言说明）
  - 检索得分：`debug_info.max_reranker_score`（如 "召回得分不足（max: 0.31）"）
  - 拒绝码：`debug_info.refuse_reason`（score_below_threshold / empty_retrieval / llm_refuse / invalid_citation / json_parse_error）

### 拒答类型说明表

Web 层需展示 LLM 主动拒答时的陷阱类型分类（`trap_type`），帮助评委理解系统检测能力：

| trap_type | 含义 | 前端展示示例 |
| --- | --- | --- |
| fake_api | 问题提及的 API/方法在 Context 中不存在 | "fake_api：文档中不存在 useLegacyAPI" |
| future_version | 问题涉及 Context 中未涵盖的未来版本特性 | "future_version：该特性在 v2.0 中才引入" |
| overgeneralization | 问题过度泛化，Context 中只有更具体的子集 | "overgeneralization：问题过于宽泛，请缩小范围" |
| parameter_mismatch | 参数名称或类型不匹配 | "parameter_mismatch：方法签名中没有 timeout 参数" |
| cross_domain | 问题跨域，如用 React 术语问 Vue 问题 | "cross_domain：问题涉及 Vue，文档为 React" |
| concept_confusion | 概念混淆，如混淆组件与 Hook | "concept_confusion：useState 是 Hook 不是组件" |
| procedure_step | 缺少操作步骤的完整上下文 | "procedure_step：缺少必要的前置步骤" |
| non_existent_attribute | 问题提及的属性/字段/选项在 Context 中不存在 | "non_existent_attribute：useState 不存在 async 属性" |

## 附录：retrieval ↔ ingestion 接口契约

> **说明**：retrieval 服务**不直连 SQLite**，通过 HTTP API 访问 ingestion 层（:3003）。向量 embedding 在 retrieval 侧本地计算（bge-m3），其余检索全部委托给 ingestion。

```
# retrieval.py → ingestion :3003

POST http://{VECTOR_API_URL}/chunks/vector-search
  body: { embedding: number[], top_k: int, filters?: dict }
  → { results: [{ chunk_id, score, content, metadata }] }

POST http://{VECTOR_API_URL}/chunks/text-search
  body: { query: string, top_k: int, filters?: dict }
  → { results: [{ chunk_id, score, bm25_rank, content, metadata }] }

GET  http://{VECTOR_API_URL}/health
  → { status: "ok", db_writable: bool, embedding_model_loaded: bool }

# retrieval 内部：retrieval ↔ rerank API（可选，RERANK_API_URL 配置）
POST http://{RERANK_API_URL}/rerank     (RERANK_API_URL 默认空，回退本地 CrossEncoder)
  body: { model: string, query: string, documents: string[], return_text: bool }
  → { results: [{ index, relevance_score, text }] }

# retrieval 内部：retrieval ↔ embedding API（可选，EMBEDDING_API_URL 配置）
POST http://{EMBEDDING_API_URL}/embeddings
  body: { model: string, input: string[] }
  → { data: [{ embedding: number[], index: int }] }
  (EMBEDDING_API_URL 默认空，回退本地 HuggingFaceEmbeddings)
```

## 附录：关键设计决策

> 本节汇总数据层（Layer 1）和检索层（Layer 2）选型决策及依据，来源为 [`backend/ingestion/README.md` §4](../src/backend/ingestion/README.md#4-关键技术决策为什么这么选) 和 [`backend/retrieval/retrieval.py`](../src/backend/retrieval/retrieval.py)。

### 选型决策汇总

#### Layer 1 决策

#### 1. SQLite + FTS5 替代 Milvus / PG / ES

| 备选 | 放弃原因 |
|---|---|
| Milvus / Qdrant | 评委环境 x86 Linux + 无外网 + Docker 单机，多进程 DB 部署成本不可控 |
| PostgreSQL + pgvector | 同上，额外运维组件 |
| Elasticsearch | 重，对评委环境压力过大 |
| **SQLite + FTS5** | ✅ 单文件、性能够用（164 文件/~5K chunks，向量全表 cosine ~100ms）、方便备份迁移、FTS5 自带 BM25 + unicode61 分词 |

**中文分词**：jieba 预切词解决 BM25 长 query / 短英文召回问题（详见 [FTS5 + jieba tokenizer spec](../src/backend/ingestion/../docs/superpowers/specs/2026-04-28-fts5-jieba-tokenizer-design.md)）。

#### 2. bge-m3 + `normalize_embeddings=True`

- 多语言（赛题中英混合），1024 维，HuggingFace 官方权重稳定
- **normalize 后 cosine 落 [0, 1]**：写入侧和查询侧都必须开，**否则 score 不可比，召回乱**
- 懒加载：进程启动不加载，首次 `/index` 调用才载（~15s），避免开发期反复重启慢

#### 3. PaddleOCR 而非 Tesseract

- 中文 OCR 准确率显著更高
- 锁版本：`paddleocr 3.x` + `paddlepaddle 3.x`（3.x 起 API 大改，降级会炸）
- GPU 部署换 `paddlepaddle-gpu` 提速 ~10x；CPU 也能跑

#### 4. .adoc 用 regex 手写而非 asciidoctor

- Spring 文档大量用 `[[xxx]]` 显式声明 anchor + `=` 系列标题，是 .adoc 子集
- 通用 markdown 解析跑 .adoc → 把 `==` 当 H2 但漏掉前一行的 `[[anchor-id]]` → **anchor 全错**
- asciidoctor / asciidoc3 库重 + 依赖 Ruby/JVM，无外网部署成本高
- 手写 regex 只识别 `=` 标题 + `[[xxx]]` 锚点，覆盖赛题 Spring .adoc 全部用法（高级语法 include / conditional / attributes 暂不支持，作普通正文）
- React / K8s 走 `markdown_parser.py`（识别 trailing `{#xxx}` / `{/*xxx*/}`）

#### 5. X1.5 section 全量化（X0 → X1.5）

- **问题**：单 chunk 切太碎，LLM 上下文不够，召回准但答不对
- **方案**：检索命中后，把同 `(file_path, title_path)` 内多个命中合并为 1 个 result，`content` 扩展为 "title prefix + 居中截 max_chars=3500" 的整段
- **效果**：+13 题增益（baseline → X1.5）
- 应急回滚：`env INGESTION_X15_ENABLED=false` 重启

#### 6. 增量索引 add / modify / delete 全部实现

- 评委评测时**当场**新增 / 修改 / 删除一篇文档，5 分钟内提问验证
- **build-once 模式直接 0 分**——必须实现增量
- MVP：`file_hash` 自动判 unchanged / replaced，走全删重写（无 embedding 复用）
- P1：升级 chunk_diff（详见 §1.4）

#### Layer 2 决策

#### 7. retrieval 通过 HTTP API 访问 ingestion，不直连 SQLite

- 检索服务（冷海军）**不直连 SQLite**，所有 DB 操作委托给 ingestion（:3003）
- 向量 embedding 在 retrieval 侧本地计算（bge-m3），避免 embedding 重复传输
- ingestion 服务暴露 `/chunks/vector-search` 和 `/chunks/text-search` HTTP 接口
- 环境变量 `VECTOR_API_URL` 配置服务地址，支持本地 / 远程切换

#### 8. reranker 采用 API / 本地 CrossEncoder 双模式

- `RERANK_API_URL` 置空 → 本地 `CrossEncoder("BAAI/bge-reranker-v2-m3")` 推理
- `RERANK_API_URL` 配置 → 远程 API（`aigw.asiainfo.com/v1/rerank`），节省本地 GPU
- 分数统一用 Sigmoid 映射到 (0,1)，保证阈值 0.4 语义一致性

#### 9. 查询扩展采用 LLM 同义 + Bing 英译三路并行

- LLM 生成最多 3 个同义变体（`QUERY_EXPANSION_NUM ≤ 5`）
- Bing Translator 自动中译英（`translators` 库），增强英文文档召回
- 每个变体独立双路检索，最后 `_merge_results` 去重合并
- 异常静默忽略（英文翻译失败不影响主流程）

## 附录：创新点

> 本节汇总系统各层的核心技术亮点与差异化设计。

### Layer 1 数据处理层

#### ① 轻量级全本地 RAG 数据栈
- 在**无外网 + Docker 单机 + x86 Linux** 评委环境下，不依赖任何外部服务
- SQLite + FTS5 单文件 DB，bge-m3 本地推理（~2GB 镜像内打包），PaddleOCR 本地 OCR
- 评委环境部署零额外依赖，一键 Docker 起服

#### ② 四 ID 锚点体系
- `file_path`：与赛题 `gold_sources[].doc_path` 完全一致，对齐判分
- `chunk_id`：`sha256(file_path | chunk_index | content[:100])`，跨进程稳定可重算
- `anchor_id`：`{file_path}#{char_offset_start}`，字符级精确定位
- `markdown_anchor`：slug 化章节锚点，**赛题判分按这个字段**，支持中文原文保留

#### ③ X1.5 section 全量化
- 突破"chunk 切太碎 → LLM 上下文不够 → 答不对"瓶颈
- 同标题下多个命中 chunk 合并为 1 个 section，title prefix + 居中截断 max_chars=3500
- **+13 题实测增益**，可一键回滚（`INGESTION_X15_ENABLED=false`）

#### ④ .adoc 精准锚点解析
- 针对 Spring .adoc 专项：手写 regex 识别 `[[anchor]]` + `=` 系列标题
- 避免通用解析器漏锚点 / 错锚点导致引用失效

### Layer 2 检索增强层

#### ⑤ retrieval → ingestion HTTP API 分离架构
- retrieval 不直连 SQLite，通过 `/chunks/vector-search` + `/chunks/text-search` 调用 ingestion
- 向量 embedding 在 retrieval 本地计算，避免 embedding 重复传输开销
- 环境变量控制服务地址，支持本地 / 远程灵活切换

#### ⑥ 三路查询扩展 + 双路并行检索
- LLM 同义变体 + Bing 英译 + 原始查询，三路扩展独立检索
- 相邻 chunk 上下文拼接后再 rerank（RERANK_CONTEXT_WINDOW=1）
- 合并去重保留双路分数（向量 score + BM25 score）

#### ⑦ reranker 双模式 + Sigmoid 归一化
- API / 本地 CrossEncoder 按环境变量自动切换
- `bge-reranker-v2-m3` logits 经 Sigmoid 映射 (0,1)，阈值 0.4 语义一致
- 相邻 chunk 上下文扩展避免 chunk 边界切断语义

### Layer 3 推理与引用层

#### ⑧ 可验证拒答机制
- `answer_status: "refused"` + `refuse_reason` 字段，拒绝理由透明
- `max_reranker_score` 随响应返回，评委可验证检索质量
- 拒答 6 大类型覆盖：虚假前提、跨域误导、诱导幻觉、越权操作请求、未来版本问题、参数不匹配

#### ⑨ 引用双验证 Pipeline
- 同步验证：正则 + LLM 双重幻觉检测
- 异步验证：URL 可访问性 + 锚点可跳转回填
- `verification_report` 字段：每条引用独立可信度评级（✓ / ? / ✗）

### 系统工程

#### ⑩ 分级 SLA + 进度可视化
- 小/中/大/超大四级 SLA 分级承诺，P2 超大文档异步处理
- P1 WebSocket 实时推送 parsing / chunking / embedding 三阶段进度

#### ⑪ MVP 全删重写 + P1 chunk_diff 原子切换
- MVP 简化实现：先删旧 chunks 再写新 chunks，无数据迁移复杂度
- P1 schema 留位：`index_version UUID` 字段 + chunk_diff 算法，为原子切换预留扩展空间

| 风险  | 概率  | 影响  | 兜底  |
| --- | --- | --- | --- |
| 网关不支持 embedding | 中   | 高   | 本地部署 bge-m3，CPU 也能跑 |
| PaddleOCR 中文识别慢 | 中   | 中   | OCR 异步，不阻塞主流程 |

| chokidar 漏事件 | 中   | 低   | 每小时全扫描 GC 兜底 |
