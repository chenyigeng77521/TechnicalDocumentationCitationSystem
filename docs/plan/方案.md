# 技术文档智能问答与引用溯源系统

> **赛题背景**
> 
>     "能否基于一批文档回答问题、且准确引用出处、不知道就说不知道"，是 Context Engineering 最基本的考查点。本题用公开技术文档作为语料，不涉及任何内部知识库。
> 
> **核心任务**
> 
> 1. 对评委会给定的文档集构建检索问答系统
> 2. 每次回答必须附带至少一条出处（文档路径 + 段落 anchor）
> 3. 对文档中不存在答案的问题，必须明确拒答而非幻觉
> 4. 支持文档增量更新（新增/修改/删除后 5 分钟内生效）
> 5. 提供 WebUI 供评委交互

## 1. 数据处理层

> 👉 负责：**“原材料对不对”**
> 
> - 文档解析错了（PDF 少页 / 乱码）
> - chunk 切错（一句话被拆烂）
> - embedding 明显异常（全是一样的向量）
> - 新文档上传后查不到
> 
> 👉 **“这玩意一开始就不对”** → 找数据层

### 1.1 文档解析

**设计**：按扩展名 + MIME sniff 路由到对应解析器，按内容类型（`document / code / structured_data`）分派后续 chunking 策略。解析器分三类：Node 内置 / Python rag-service / 转格式后再解析。

#### 1.1.1 支持格式与解析器路由

**文档类** ：

| 扩展名 | 解析器 | 处理方式 | 优先级 |
| --- | --- | --- | --- |
| `.md` | Node 内置（`marked`）✅ | 直读  | P0  |
| `.txt` | Node 内置 | 直读，按扩展名推测编码 | P0  |
| `.html` | Node 内置（`cheerio` + html2md） | 转 markdown 后走 md 路径 | P0  |
| `.pdf`（文字版） | `Unstructured`（Python） | 通过 rag-service | P0  |
| `.pdf`（扫描版） | `PaddleOCR`（Python） | 通过 rag-service，中文强 | P1  |
| `.docx` | `mammoth`（Node）→ md | 现有实现可用，P1 升级 Unstructured | P0  |
| `.doc`（旧版） | LibreOffice 转 .docx → mammoth | 需要额外依赖 | P1  |
| `.xlsx` | `xlsx`（Node） | 按 sheet 转 markdown 表格 | P0  |
| `.xls`（旧版） | LibreOffice 转 .xlsx → xlsx | —   | P1  |
| `.pptx` | `Unstructured`（Python） | 每 slide 一个段落 | P1  |
| `.ppt`（旧版） | LibreOffice 转 .pptx → pptx 处理 | —   | P1  |

**代码类** ：

| 扩展名 | 语言  | chunking 特殊规则 |
| --- | --- | --- |
| `.py` | Python | 按 `^class` / `^def` / `^async def` 切 |
| `.java` | Java | 按 class / method 切 |
| `.cpp` / `.hpp` / `.h` | C/C++ | 按函数定义切 |
| `.sql` | SQL | 按 `;` + 空行切 |
| `.json` | JSON | 小文件 1 chunk；大文件按顶层 key 切 |
| `.yaml` / `.yml` | YAML | 按顶层 key 切；保留注释 |
| `.xml` | XML | 按根 tag 切 |
| `.html`（代码视角） | HTML | 含 `<body>` 布局 → 文档；否则 → 代码 |

#### 1.1.2 统一输出格式

```typescript
    pdf_pages?: number;
    pdf_is_scanned?: boolean;

  };

```

**HTTP 接口**：

```

Content-Type: multipart/form-data
  file: <binary>
  hint_language: "zh" | "en" | "auto"   (optional)
→ ParseResult
```

#### 1.1.3 关键决策

- ✅ **Node 处理 md/txt/html/docx/xlsx**（现有能力复用）
- ✅ **Python 处理 pdf/pptx/代码 AST/旧版 Office**（生态强）
- ✅ **LibreOffice 统一旧版 Office 转换**（避免维护多套二进制解析器）
- ✅ **MIME sniff 兜底**：用 `file-type`（Node）/ `python-magic`（Python）二次确认真实类型
- ✅ **代码不加 OCR**（源码不可能是扫描件）

### 1.2 Chunk 切分策略

按 `content_type` 分派到三套切分策略，统一走 Node 模块。

#### 1.2.1 document 类型：段落 → 句子 → 硬切 三级 fallback

适用于 md / pdf / docx / pptx / html / txt 转出的 markdown：

```
第 1 级：按标题（title_tree）切成大段
   │  段落 ≤ MAX_CHARS → 直接作为 chunk
   ▼
第 2 级：按段落边界（\n\n）切分
   │  段落 ≤ MAX_CHARS → 直接作为 chunk
   ▼
第 3 级：按句子边界（。！？. ! ?）切分
   │  单句仍 > MAX_CHARS → 触发硬切（is_truncated: true）
   ▼
添加 200 char overlap（取前一 chunk 末尾拼到下一 chunk 前面）
```

#### 1.2.2 code 类型：语言感知的结构化切分

```
第 1 级：按语言的"顶级定义"切（class / def / function / CREATE 等）
   │  block ≤ MAX_CHARS → 直接作为 chunk，函数名/类名作为 title_path
   ▼
第 2 级：超长函数按空行切逻辑块
   │  逻辑块仍 > MAX_CHARS → 触发硬切（极少见）
   ▼
注意：代码 chunk 不加 overlap（语法上会导致重复）
```

各语言顶级定义匹配规则：

| 语言  | 切分边界 |
| --- | --- |
| Python | `^class` / `^def` / `^async def` |
| Java | `public/private/protected … class/interface/enum` |
| C/C++ | `class … {` / 函数定义 |
| SQL | `CREATE / SELECT / INSERT / UPDATE / DELETE` |
| 未知语言 | 整文件一个 block（兜底） |

#### 1.2.3 structured_data 类型：按顶层 key / 节点切

- 文件 ≤ MAX_CHARS → 整体 1 chunk
- JSON / YAML → 按顶层 key 切（每个 key 独立 chunk，title_path 用 key 路径）
- XML → 按根 tag 的直接子节点切
- 解析失败 → 降级为纯文本硬切

#### 1.2.4 Chunk 数据结构

```typescript
interface Chunk {
  chunk_id: string;        // sha256(file_path + chunk_index + content[:100])
  file_path: string;
  file_hash: string;
  content: string;
  anchor_id: string;       // "file_path#char_offset_start"
  title_path: string | null;
  char_offset_start: number;
  char_offset_end: number;
  char_count: number;      // 用 char 数代替 token 数（避免 tokenizer 不对齐）
  is_truncated: boolean;   // 硬切时 true，Layer 3 在 prompt 中注明
  chunk_index: number;
  content_type: "document" | "code" | "structured_data";
  language?: string;
  embedding?: number[];
}
```

**Anchor 设计（双字段，职责分离）：**

- `anchor_id`（主键，程序定位用）：`file_path#char_offset_start`，例如 `docs/api/auth.md#4821`
- `title_path`（可读性附属字段，UI 展示用）：`Authentication > OAuth2 > Token Refresh`

对于无标题的文档（纯文本、配置文件等），`title_path` 留空，仅使用 `anchor_id`，系统不因此降级。

**各类型关键决策汇总：**

| 决策项 | document | code | structured_data |
| --- | --- | --- | --- |
| 切分粒度 | 段落/句子/硬切 | 顶级定义/逻辑块 | 顶层 key |
| overlap | 200 char | **无** | 无   |
| title_path 来源 | heading 路径 | 函数名/类名 | key 路径 |
| 硬切 fallback | ✓（罕见） | ✓（极少） | ✗（按 parser） |
| 质量过滤（< 30 char） | ✓   | ✗（短函数也有意义） | ✗   |
| 保留原始格式 | 允许归一化 | **严格保留** | **严格保留** |

**Schema**：

```sql
CREATE TABLE documents (

    indexed_at       TIMESTAMP,
    created_at       TIMESTAMP DEFAULT CURRENT_TIMESTAMP

CREATE TABLE chunks (
    chunk_id          TEXT PRIMARY KEY,
    file_path         TEXT NOT NULL,
    file_hash         TEXT NOT NULL,
    index_version     TEXT NOT NULL,
    content           TEXT NOT NULL,
    anchor_id         TEXT NOT NULL,
    title_path        TEXT,

    char_offset_end   INTEGER NOT NULL,

    FOREIGN KEY (file_path) REFERENCES documents(file_path) ON DELETE CASCADE
);

CREATE INDEX idx_chunks_version ON chunks(index_version);

    chunk_id UNINDEXED,
    content,
    title_path,
    tokenize = 'unicode61 remove_diacritics 2'
);

-- 自动同步触发器
CREATE TRIGGER chunks_ai AFTER INSERT ON chunks BEGIN
    INSERT INTO chunks_fts(chunk_id, content, title_path)
    VALUES (new.chunk_id, new.content, new.title_path);
END;

CREATE TRIGGER chunks_ad AFTER DELETE ON chunks BEGIN
    DELETE FROM chunks_fts WHERE chunk_id = old.chunk_id;
END;
```

### 1.4 增量同步

**设计**：双路径 → 单 Pipeline → 文件级锁 → chunk 级 diff → 原子版本切换

增量触发来源有两条路径，统一合并到同一 Pipeline：

```
路径 A：WebUI 上传
  POST /api/upload 接收文件 → 保存到 /data/docs/ → 直接调用 handle_file_event()

路径 B：外部文件变更（评委直接拷贝文件到目录）
  chokidar 监听 /data/docs/ → debounce 1s → 调用 handle_file_event()
```

`index_pipeline` 完整流程：

```python
async def index_pipeline(file_path: str):
    # 1. 计算新 hash；hash 未变直接返回
    new_hash = sha256_of(file_path)
    old_record = db.get_document(file_path)
    if old_record and old_record.file_hash == new_hash:
        return { "status": "unchanged" }

    # 2. 生成新 index_version（原子切换开关）
    new_version = str(uuid.uuid4())
    db.upsert_document(file_path, hash=new_hash,
                       version=new_version, status='pending')

    try:
        # 3. 解析 + 切分
        parse_result = await parse_document(file_path)
        new_chunks   = chunker.split(parse_result)

        # 4. chunk 级 diff
        old_chunks = db.get_chunks(file_path,
                         version=old_record.index_version) if old_record else []
        to_add, to_delete, reuse = chunk_diff(new_chunks, old_chunks)

        # 5. 复用未变 chunk 的 embedding
        for new_chunk, old_chunk in reuse:
            new_chunk.embedding = old_chunk.embedding

        # 6. 只对 to_add 调用 embed（省 99% embedding 调用）
        embeddings = await batch_embed(
            [c.content for c in to_add],
            concurrency=8,
            rate_limit=gateway_qps - 2
        )
        for chunk, emb in zip(to_add, embeddings):
            chunk.embedding = emb

        # 7. 先写新 version → 再删老 version（失败时老数据保留）
        db.insert_chunks([{**c, "index_version": new_version}
                          for c in new_chunks])
        db.update_document(file_path, status='indexed',
                           version=new_version,
                           chunk_count=len(new_chunks),
                           indexed_at=now())
        if old_record:
            db.delete_chunks_by_version(file_path,
                                        old_record.index_version)

    except ParseError as e:
        db.update_document(file_path, status='error',
                           error_detail=f"解析失败: {e.type}，请确认文件未加密或格式正确")
    except Exception as e:
        db.update_document(file_path, status='error', error_detail=str(e))
```

**chunk_diff 算法**：

```python
def chunk_diff(new_chunks, old_chunks):
    new_ids = {c.chunk_id: c for c in new_chunks}
    old_ids = {c.chunk_id: c for c in old_chunks}

    to_add    = [c for id, c in new_ids.items() if id not in old_ids]
    to_delete = [c for id, c in old_ids.items() if id not in new_ids]
    reuse     = [(new_ids[id], old_ids[id]) for id in new_ids if id in old_ids]

    return to_add, to_delete, reuse
```

**删除处理**：

```python
async def handle_file_delete(file_path: str):
    async with file_lock(file_path):
        db.delete_document(file_path)  # ON DELETE CASCADE 删所有 chunks
```

**孤儿 GC**（启动时 + 每小时）：

```python
async def gc_orphan_chunks():
    for fp in db.get_all_document_paths():
        if not os.path.exists(fp):
            db.delete_document(fp)
    db.execute("""
        DELETE FROM chunks
        WHERE file_path NOT IN (SELECT file_path FROM documents)
    """)
```

**关键决策：**

- ✅ **文件级锁**：`async-mutex`（Node）/ `asyncio.Lock`（Python），单进程够用，不需要 Redis 分布式锁
- ✅ **debounce 1 秒**：抑制文件事件风暴（快速连续写入合并为一次）
- ✅ **chunk_diff 复用 embedding**：大文档改一段时省 99% embedding 调用
- ✅ **原子版本切换**：查询永远看到一致的版本，失败时保留老数据
- ✅ **定时 GC**：防漏事件导致孤儿 chunks 堆积

### 1.5 5 分钟 SLA 保障

**分级 SLA + 动态并发 + WebSocket 进度可视化**

| 文档大小 | 解析  | 切分  | Embedding | 合计  | SLA 级别 |
| --- | --- | --- | --- | --- | --- |
| 小（< 20 页 / 100KB） | 1-5s | 1s  | 5-10s（~20 chunks × 8 并发） | **< 30s** | P0 必达 |
| 中（20-100 页） | 5-30s | 2-5s | 20-60s | **< 2min** | P0 必达 |
| 大（100-500 页） | 30s-2min | 5-15s | 1-3min | **< 5min** | P1 尽力 |
| 超大（> 500 页 / 50MB+） | 2-10min | 10-30s | 3-10min | **异步 + 进度条** | P2 不承诺 SLA |

**动态并发控制（rate limit 感知）：**

```python
class EmbedPool:
    def __init__(self, gateway_qps: int):
        self.sem = asyncio.Semaphore(min(8, gateway_qps - 2))

    async def embed(self, text: str) -> list[float]:
        async with self.sem:
            return await self._embed_with_retry(text)

    @retry(wait=wait_exponential(min=1, max=60),
           stop=stop_after_attempt(5),
           retry=retry_if_exception_type(RateLimitError))
    async def _embed_with_retry(self, text: str):
        return await gateway.embed(text)
```

## 2. 检索增强层

> 👉 负责：**“找没找对东西”**
> 
> - 明明文档里有，但就是搜不到
> - 搜出来一堆不相关内容
> - 排序很离谱（垃圾排第一）
> - 拒答阈值误判（该答不答）
> 
> 👉 **“仓库里有货，但你没拿对” → 找检索层**

### 2.1 查询扩展（Query Expansion）

**问题背景：** bge-m3 在以下场景存在语义理解局限：

- **专业术语漂移**：`RAFT`、`gRPC`、`CRD` 等技术专有名词的向量表示可能和通用语义混淆，导致召回偏离
- **短查询语义稀疏**：评委提问往往很短（如"超时配置"），短文本 embedding 向量信息密度低，召回方差大
- **中英混合对齐偏差**：技术文档中文描述 + 英文术语混排，跨语言语义对齐得分偏低

**解决方案：** 向量检索前对原始查询做轻量扩展，增强语义覆盖面：

```python
def expand_query(query: str) -> str:
    """调用 LLM 生成同义表达，拼接后统一 embed，token 消耗极小"""
    expansions = llm.generate(
        f"给出以下技术问题的2个同义表达，仅输出问题本身，换行分隔，不加序号：\n{query}"
    )
    return query + " " + " ".join(expansions.strip().split("\n"))

# 示例：
# 原始查询：  "超时配置"
# 扩展后：    "超时配置 连接超时设置 timeout configuration"
```

### 2.2 两路召回 + RRF 融合

两路召回在 Node 服务内完成，手动实现 RRF 融合：

**路 1：BM25（SQLite FTS5）**

```sql
SELECT chunk_id, rank
FROM   chunks_fts
WHERE  chunks_fts MATCH ?    -- expanded_query
ORDER  BY rank
LIMIT  50;
```

**路 2：Dense kNN（全表余弦相似度）**

```sql
SELECT chunk_id,
       vec_cosine_similarity(embedding, ?) AS score
FROM   chunks
WHERE  embedding IS NOT NULL
ORDER  BY score DESC
LIMIT  50;
```

> 50MB 语料（估 ~10k chunks）单次全表 cosine 约 100ms，可接受。P1 可升级 `sqlite-vec` 扩展，查询提速 10x。

**RRF 融合（Node 实现）：**

```typescript
function rrf(bm25Ranks: Map<string, number>,
             denseRanks: Map<string, number>,
             k = 60): ScoredChunk[] {
  const allIds = new Set([...bm25Ranks.keys(), ...denseRanks.keys()]);
  const scores = new Map<string, number>();

  for (const id of allIds) {
    const r1 = bm25Ranks.get(id)  ?? Infinity;
    const r2 = denseRanks.get(id) ?? Infinity;
    scores.set(id, 1 / (k + r1) + 1 / (k + r2));
  }

  return [...scores.entries()]
    .sort((a, b) => b[1] - a[1])
    .slice(0, 20)
    .map(([id, score]) => ({ chunk_id: id, rrf_score: score }));
}
```

```python
def get_top_k(query: str) -> int:
    BROAD_KEYWORDS  = ["所有", "列举", "比较", "对比", "区别", "有哪些", "分别"]

    if any(kw in query for kw in BROAD_KEYWORDS):
        return 8   # 综合型问题，扩大召回
    if any(kw in query for kw in SIMPLE_KEYWORDS):
        return 3   # 事实型问题，精确召回
    return 5       # 默认值
```

> 👉 负责：**“说没说人话 / 有没有胡编”**

> - 明明给了上下文却答错

### 3.1 上下文注入格式

每个 Chunk 包装为：


### 3.2 提示词

```
你是一个严格的技术文档问答助手。

1. 仅根据下方提供的 Context 回答，严禁使用任何外部知识。
```

### 3.3 引用验证 Pipeline（分级，不阻塞响应）

**同步验证（< 10ms，阻塞响应链路）：**

检查 LLM 输出中引用的所有 `[n]` ID 是否真实存在于本次检索到的 Chunk 列表。如发现无效 ID，自动剔除该引用标记并在响应中注明。

**异步验证（不阻塞，后台执行）：**

对回答中的关键名词、数字、版本号进行 token 级匹配，验证其是否出现在对应 Chunk 原文中。结果写入质量日志，在 UI 引用旁异步回填可信度标记：

- `✓`：关键内容在原文中找到支撑
- `?`：匹配不确定，建议核查原文

```python
async def verify_citations_async(answer: str, chunks: list[Chunk]):
    for citation_id, claim in extract_claims(answer):
        chunk = chunks[citation_id]
        key_tokens = extract_key_tokens(claim)   # 名词、数字、版本号
        matched = all(tok in chunk.raw_text for tok in key_tokens)
        await db.write_citation_quality(citation_id, matched)
        await ws.push_ui_update(citation_id, "✓" if matched else "?")
```

## 附录：风险与兜底

| 风险  | 概率  | 影响  | 兜底  |
| --- | --- | --- | --- |
| 网关不支持 embedding | 中   | 高   | 本地部署 bge-m3，CPU 也能跑 |
| PaddleOCR 中文识别慢 | 中   | 中   | OCR 异步，不阻塞主流程 |
| SQLite 50MB 后性能降 | 低   | 中   | 升级 sqlite-vec |
| chokidar 漏事件 | 中   | 低   | 每小时全扫描 GC 兜底 |
| 大文档 SLA 超时 | 高   | 中   | 分级 SLA + 异步承诺 + 进度条 |
| chunk 切分边界错乱 | 低   | 高   | 评测集验证 + 三级 fallback |
| LibreOffice 转换失败 | 中   | 低   | try-catch 记 error_detail，降级为 antiword / catdoc 纯文本提取 |
| 代码文件编码非 UTF-8 | 中   | 中   | chardet 检测 + 转码；失败标记 error |
| 代码"顶级定义"regex 误判 | 低   | 低   | 整文件当一个 chunk 兜底 |
| 二义性文件判错类型 | 中   | 低   | UI 提供"手动重新分类"按钮 |
